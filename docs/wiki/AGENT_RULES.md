# 🤖 Правила для AI-Агентов (Agent Rules)

> [!IMPORTANT]
> Если ты (Claude, Cursor, Antigravity) читаешь этот файл, это значит, что ты находишься в режиме Planning Mode или выполняешь задачу по модификации кодовой базы проекта OCR. **ОБЯЗАТЕЛЬНО соблюдай все перечисленные ниже правила.**

## 1. Сохранение знаний в LLM Wiki

Платформа использует концепцию **LLM Wiki** (паттерн Андрея Карпаты). 
При добавлении нового профиля обработки, изменении архитектуры или правил деплоя, ты **ОБЯЗАН** обновить соответствующие файлы в директории `docs/wiki/`. 

Никакие знания не должны оставаться только в чате или в истории коммитов.

## 2. Строгие архитектурные слои

При написании или рефакторинге кода в `src/ocr_platform` соблюдай зависимости:
- `api` имеет доступ только к `orchestration` и `observability`. Никаких прямых вызовов БД!
- `services` реализует бизнес-логику и имеет доступ к `storage`, `observability` и `config`.
- `storage` ничего не знает про бизнес-логику и маршрутизацию.
- Не "зашивай" (hardcode) бизнес-правила (например, как извлекать номер дела) прямо в код `orchestration`. Вся логика должна настраиваться в YAML-профилях в `config/pipelines/`.

*Подробнее читай в [ARCHITECTURE.md](file:///c:/Users/user2/Documents/Cursor/OCR/docs/wiki/ARCHITECTURE.md).*

## 3. Контракты API и Схемы (Pydantic)

- Все входные и выходные данные должны быть описаны в `src/ocr_platform/api/schemas.py`.
- Принимаем файлы через `POST /documents/ingest`. Если в запросе не передан тип документа (`document_type`), не пытайся угадать его в коде через `if/else`. Оркестратор обязан использовать LLM для определения типа (см. профиль `system/router.yaml`).

## 4. Логирование и Метрики

Каждый новый сервис или шаг пайплайна **ОБЯЗАН**:
- Писать структурированные JSON-логи с указанием `document_id` и `pipeline_run_id`.
- Обновлять метрики в `src/ocr_platform/observability/metrics.py`.
- Если это ML-шаг или LLM-классификатор, отправлять данные в **MLflow** через `mlflow_client` (логировать модель, уверенность, параметры).

## 5. Запреты

- ❌ Не добавляй эвристики для определения типа документа. Только LLM.
- ❌ Не модифицируй существующие YAML профили (например, `court_decision_ru.yaml`), если задача только "поправить баг". Если нужно добавить новый документ, создавай новый YAML файл.
- ❌ Не пиши SQL-запросы вручную, используй SQLAlchemy ORM в слое `storage`.

## 6. Как выполнять задачи (Workflow)

1. Прочитай актуальные цели в `PLANS.md` (и детальные планы в `docs/exec-plans/`, если они там есть).
2. Для каждой задачи:
   - Уточни профиль/тип документа и шаги пайплайна в YAML.
   - Обновляй код только в допустимых архитектурных слоях.
   - Добавь или обнови тесты.
3. Убедись, что:
   - Тесты и линтеры проходят.
   - Новые шаги пишут структурные JSON-логи, отправляют метрики и события в MLflow.
4. **Важное правило**: Если документации не хватает, сначала предложи обновление в Wiki (папка `docs/wiki/`), и только потом обновляй код по новым правилам.

## 7. Локальная отладка без Docker (Fast Feedback Loop)

При внесении изменений в логику пайплайнов (особенно в агенты извлечения на базе `pydantic-ai`), пересборка Docker-контейнеров может занимать слишком много времени. 
Для ускорения отладки (Fast Feedback Loop) используй локальный запуск:
1. Выполни `pip install -e .[nlp]` в локальном окружении Windows, чтобы синхронизировать зависимости и установить проект как редактируемый пакет.
2. Запускай тестовые скрипты напрямую (например, через `python src/test_script.py` или `pytest`), полностью минуя очереди RabbitMQ и FastAPI-оркестратор. Это позволит получить ответ от LLM за секунды, а не минуты.
3. После того как логика отлажена и стабильна локально, можно пересобирать Docker-образы для финального интеграционного тестирования.

## 8. Версионность библиотек (Осторожно с PydanticAI)

В проекте жестко зафиксирована версия `pydantic-ai==0.0.18`. 
API этой библиотеки подвержено серьезным изменениям между минорными версиями. 
При работе с агентами извлечения (`extraction_agent.py`) строго соблюдай следующие правила API версии 0.0.18:
- Модель инициализируется напрямую через `OpenAIModel("model_name", base_url=..., api_key=...)` (без использования класса `OpenAIProvider`, который появился в более поздних версиях).
- При инициализации `Agent`, тип возвращаемого результата задается через параметр `result_type=` (а не `output_type=`).
- Результат работы агента (`RunResult`) содержит данные в свойстве `result.data` (а не `result.output`). 

## 9. Доменная специфика (ФНС и налоговые органы)

При обработке судебных заявлений кредиторов (РТК), документы от налоговых органов (ФНС) имеют нестандартный формат шапки и часто не содержат прямого указания ИНН. 
В коде реализована отдельная логика обработки:
- Система ищет регулярные выражения с упоминанием `"МИНФИН РОССИИ ФЕДЕРАЛЬНАЯ НАЛОГОВАЯ СЛУЖБА"`, `"МЕЖРАЙОННАЯ ИНСПЕКЦИЯ"` или `"ФЕДЕРАЛЬНОЙ НАЛОГОВОЙ СЛУЖБЫ"`.
- При обнаружении этих фраз используется специализированный LLM-агент для извлечения наименования кредитора из шапки.
- ИНН кредитора затем восстанавливается через автоматический веб-поиск (DuckDuckGo/HTML-поиск) по найденному названию инспекции, чтобы предотвратить галлюцинации LLM.
