# 🏛 Архитектура OCR-платформы

Проект построен по модульному принципу и разделен на два ключевых компонента, которые тесно взаимодействуют друг с другом:

1. **`OCR-document-parser`** — основная платформа, оркестратор и OCR-движок на FastAPI.
2. **`NLP-entity-extractor`** — локальный NLP/NER микросервис для интеллектуального анализа (полностью интегрирован в логику парсера).

Цель платформы — построение декларативных пайплайнов обработки документов (в т.ч. решений судов) с возможностью принимать файлы, хранить версии текстов, извлекать сущности, оценивать качество и логировать все метрики.

Стек: **Python + FastAPI + PostgreSQL + MLflow + HuggingFace (PyTorch)**.

---

## 1. Слои и Модули (OCR-document-parser)

Код оркестратора расположен в пакете `src/ocr_platform` и строго разделен на следующие слои:

### 1.1. API (`src/ocr_platform/api`)
**Ответственность:**
- HTTP-контракты (FastAPI endpoints).
- Базовые эндпоинты:
  - `POST /documents/ingest` — принять документ и вернуть `document_id`.
  - `GET /documents/{id}/result` — получить результаты извлечения.
  - `GET /human-review/tasks` и `POST /human-review/tasks/{id}/submit` — работа с ручной верификацией.
- Валидация входа/выхода через Pydantic-схемы (описаны в `src/ocr_platform/api/schemas.py`).
- Преобразование сетевых DTO во внутренние объекты.

### 1.2. Оркестратор (`src/ocr_platform/orchestration`)
**Ответственность:**
- Определение профиля обработки по типу документа (Routing).
- Исполнение шагов пайплайна согласно YAML-конфигурации.
- Запись событий пайплайна (start/finish/error) в базу данных.

### 1.3. Сервисы (`src/ocr_platform/services`)
**Ответственность:**
- Бизнес-логика и вызовы ML/AI сервисов.
- **`ocr_service.py`**: Извлечение текста (pdfplumber) и запуск OCR (DeepSeek OCR, Tesseract).
- **`document_intel_service.py`**: Запуск NLP-экстрактора для получения структурированных полей.
- **`validation_service.py`**: Проверка форматов (ИНН, даты, суммы) и согласованности.
- **`quality_service.py`**: Подсчет Quality Score.

### 1.4. Хранилище (`src/ocr_platform/storage`)
**Ответственность:**
- Доступ к базе данных PostgreSQL (SQLAlchemy ORM-модели). Основные таблицы:
  - `documents`, `document_files`, `pipeline_runs`, `pipeline_events`
  - `text_versions`, `structured_versions`, `quality_scores`
  - `human_review_tasks`, `human_review_actions`
- Файловое хранилище (сохранение оригиналов PDF/изображений).

### 1.5. Наблюдаемость (`src/ocr_platform/observability`)
**Ответственность:**
- `logging.py` — настройка структурных логов (JSON).
- `metrics.py` — экспорт метрик (Prometheus/StatsD).
- `mlflow_client.py` — агрегация логов, метрик и артефактов (например, `document_type_attempts.json`) и отправка данных в **MLflow**.

## 2. Локальный NLP-экстрактор (NLP-entity-extractor)

Этот микросервис отвечает за интеллектуальный анализ русскоязычного текста.

**Принцип работы:**
- Базируется на предобученной модели `Gherman/bert-base-NER-Russian`.
- Выполняет поиск ФИО, названий судов, ИНН и дат.
- Использует библиотеку `pymorphy3` для нормализации ФИО (приведение к именительному падежу).
- В продакшене его веса загружаются прямо в воркер `OCR-document-parser`, поэтому отдельный процесс для него не поднимается. Он вызывается как библиотека.

---

## 3. Правила зависимостей слоев

Для сохранения чистоты архитектуры действуют строгие правила (направление сверху вниз):

✅ **Разрешено:**
- `api` → `orchestration`, `observability`.
- `orchestration` → `services`, `storage`, `config`.
- `services` → `storage`, `observability`, `config`.

❌ **Запрещено:**
- `storage` вызывать `api`, `orchestration`, `services`.
- Пересечение слоев снизу вверх (например, вызов HTTP-роутов прямо из `services`).

## 4. Декларативная Конфигурация (YAML)

Все пайплайны и бизнес-процессы описываются в файлах конфигурации в `src/ocr_platform/config/pipelines/`.

- `system/router.yaml` — логика определения типа документа (маршрутизация). Включает fallback-цепочку LLM для классификации.
- `profiles/court_decision_ru.yaml` — профиль "Решение суда". Задает шаги пайплайна, выбор экстрактора (`nlp` или `llm`), требуемые поля и пороги качества.
- `profiles/rtk.yaml` — профиль "Заявление о включении в РТК". Задает шаги пайплайна и поля для реестра требований кредиторов.

*Детали работы конфигурации, выбор экстрактора (LLM vs NLP) и список моделей описаны в [COMPONENTS.md](/docs/wiki/COMPONENTS.md).*
