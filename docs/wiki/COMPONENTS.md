# 🧩 Компоненты и Пайплайны

В этом документе описывается, как работают ключевые модули извлечения текста и маршрутизации документов в платформе, а также какие поля мы умеем доставать.

## 1. Пайплайны (Pipeline Engine)

Вся обработка документов является **декларативной**. Когда документ попадает в систему, оркестратор (`orchestration/pipeline_engine.py`) читает его и определяет профиль обработки.

### 1.1 Роутинг и Классификация (system/router.yaml)
Если пользователь загрузил файл, но не указал тип документа, оркестратор запускает LLM-классификатор (через OpenAI/OpenRouter API). **Эвристики для определения типа документа не используются, только LLM.**

1. LLM анализирует извлеченный текст.
2. Возвращает JSON с полем `document_type` и `confidence`.
3. Если уверенность ниже `min_confidence` (указанного в router.yaml), документу принудительно присваивается статус `unknown`.
4. В противном случае `document_type` сопоставляется с одним из профилей (например, `court_decision_ru.yaml` или `rtk.yaml`).

> [!NOTE]
> В базу данных всегда пишется поле `detection_source`, которое указывает, как был определен тип:
> - `request` (передан напрямую при загрузке)
> - `llm` (определен через LLM)
> - `low_confidence_fallback` (уверенность LLM ниже порога)
> - `llm_unavailable` / `llm_no_text` (fallback на `unknown`)

### 1.2 Выполнение Профиля
Каждый профиль содержит:
- **`applicable_sources`**: источники (crm, email, api).
- **`pipeline`**: список шагов (extract_entities → validate → compute_quality).
- **Выбор экстрактора**: ключ `extractor` (со значениями `llm` или `nlp`), который определяет движок для извлечения полей.
- **`fields_llm`** / **`fields_nlp`**: схема полей, которые нужно извлечь.

## 2. Типы документов и извлекаемые поля

Сейчас в системе настроены два основных профиля документов. Для каждого профиля задан строгий набор извлекаемых полей:

### 2.1 Решение суда (`court_decision_ru.yaml`)
- **ФИО должника** (debtor_full_name)
- **ИНН должника** (debtor_inn)
- **Номер дела** (case_number)
- **ФИО судьи** (judge_full_name)
- **Название суда** (court_name)
- **Дата решения** (decision_date)
- **Тип процедуры** (procedure_type) — всегда возвращается в **именительном падеже** (например: `"процедура реализации имущества гражданина"`, а не `"процедуры реализации..."`)
- **Дата окончания процедуры** (procedure_end_date) — *опционально*
- **Вычислена ли дата** (procedure_end_date_is_calculated) — *опционально*
- **Заблаговременное предоставление отчета ФУ** (early_report_deadline) — *опционально*
- **Мотивирующая часть** (motivating_part) — *опционально*
- **Резолютивная часть** (resolutive_part) — *опционально*

### 2.3 Паспорт гражданина РФ (`passport_main.yaml` и `passport_registration.yaml`)

Паспорт разделен на два самостоятельных профиля в зависимости от разворота:

**1. Главная страница / разворот** (`passport_main.yaml`):
- **Серия паспорта** (passport_series): 4 цифры
- **Номер паспорта** (passport_number): 6 цифр
- **Фамилия** (last_name)
- **Имя** (first_name)
- **Отчество** (patronymic) — опционально
- **Пол** (gender): Мужской / Женский
- **Дата рождения** (birth_date)
- **Место рождения** (birth_place)
- **Дата выдачи** (issue_date)
- **Код подразделения** (department_code): формат XXX-XXX
- **Кем выдан** (issued_by)

**2. Страница с пропиской** (`passport_registration.yaml`):
- **Серия паспорта** (passport_series): 4 цифры — опционально
- **Номер паспорта** (passport_number): 6 цифр — опционально
- **Адрес регистрации** (registration_address)

Извлечение выполняется соответствующим комбинированным агентом (`agent_passport_main_combined` или `agent_passport_registration_combined`). При вызове с legacy-типом `passport` система автоматически определяет профиль с учетом `page_type`.

2.2 Заявление о включении в РТК (`rtk.yaml`)
- **Кредитор** (creditor): Наименование юридического или ФИО физического лица.
- **ИНН кредитора** (creditor_inn): ИНН кредитора (10 или 12 цифр).
- **Сумма требований** (claims_amount): Общая сумма задолженности, указанная в требованиях.
- **Основания** (grounds): Канонические основания возникновения задолженности (кредитный договор, договор займа и т.д.).
- **Номер дела** (case_number): Номер дела Арбитражного суда в формате `А05-6/2025`.

**Особенности для налоговых документов (ФНС):**
- Система автоматически определяет документы ФНС по ключевым фразам (МИНФИН, МЕЖРАЙОННАЯ ИНСПЕКЦИЯ, ФЕДЕРАЛЬНОЙ НАЛОГОВОЙ СЛУЖБЫ) в начале текста.
- **Порядок экстракции**: сначала `creditor`, потом `creditor_inn` (обратный порядок).
- **Кредитор**: извлекается `creditor_header` (наименование и номер межрайонной инспекции) через специализированный SGR-промпт.
- **ИНН кредитора**: принудительно ищется через веб-поиск (`search_creditor_inn`) по имени межрайонной инспекции для исключения галлюцинаций LLM.

**Особенности для не-налоговых документов РТК:**
- **Объединенный LLM-запрос (Оптимизация токенов)**: Вместо трех раздельных запросов для `creditor_inn`, `claims_amount` и `grounds`, система выполняет **один комбинированный запрос** через специализированного агента `agent_rtk_combined` с выходным форматом `RtkCombinedResult`. Текст документа передается в LLM только один раз, а извлеченные поля сохраняются в кэше результатов и пропускаются в последующем цикле экстракции.
- **Порядок экстракции**: сначала извлекается комбинированная тройка (`creditor_inn`, `claims_amount`, `grounds`), а затем `creditor`.
- **Автоматическая верификация кредитора по ИНН**: 
  - Если в комбинированном ответе найден корректный ИНН кредитора (10 или 12 цифр), система запускает веб-поиск по ИНН (`search_creditor_name`).
  - Результаты поиска передаются во вспомогательного агента (`company_name_extraction_agent`) со схемой `CompanyNameResult`.
  - Агент вытягивает официальное наименование организации непосредственно из поисковой выдачи, приводя его к стандартной форме (например, "ООО «АЙДИ КОЛЛЕКТ»") и очищая от лишних символов. Это исключает опечатки OCR при чтении имени кредитора из документа.
  - Если ИНН отсутствует, система падает на резервную LLM-экстракцию кредитора по полному тексту документа.
- **Резервный Regex-поиск ИНН**: если LLM не смогла найти ИНН, применяется резервный регулярный поиск ИНН в тексте не-налогового документа.

## 3. Модуль Извлечения Текста и OCR (ocr_service.py)

Служит "входной дверью" для любых файлов. Выполняется один раз в начале обработки.

1. **PDF с текстовым слоем**: Парсится через библиотеку `pdfplumber`. Это самый быстрый и точный метод.
2. **Сканированные PDF или Картинки**: Если текстовый слой отсутствует, запускается OCR-движок. В системе реализован **каскадный механизм (fallback)**, который управляется через переменную окружения `OCR_ENGINE`:
   - **RouterAI Gemini OCR** (`engine="router_ai"`): Запрос отправляется на API RouterAI (модель `google/gemini-2.5-flash`) для высококачественного распознавания PDF (передаются целиком) или изображений.
   - **Tesseract OCR** (`engine="tesseract"`): Легкий локальный OCR (tesseract-ocr-rus). Является движком по умолчанию.

### 3.1. RouterAI Gemini OCR

Обработка документов происходит путем передачи файлов напрямую в Gemini 2.5, который нативно поддерживает PDF и большие контексты.

Логика отката:
- Ошибка RouterAI (напр. таймаут или недоступность API) → **Tesseract** (локальный OCR)

### 3.2. Кэширование распознанного текста (TextVersion)

Для предотвращения дорогого повторного OCR при перезапуске упавших пайплайнов (например, из-за сбоев внешних API на шаге экстракции), в оркестраторе реализовано кэширование:
- При запуске обработки документа система проверяет наличие сохраненной записи `TextVersion` в базе данных.
- Если запись найдена и содержит непустой текст, OCR и парсинг PDF полностью пропускаются, а текст мгновенно восстанавливается из базы.
- Во избежание дублирования данных при повторных попытках выполнения того же `pipeline_run_id` существующая запись `TextVersion` обновляется, а не создается заново.

## 6. Декларативные SGR-промпты и типизированное извлечение

Вместо хардкодных функций на Python логика интеллектуального анализа полностью перенесена в YAML-профили документов (`rtk.yaml` и `court_decision_ru.yaml`) в виде SGR-инструкций (Step-by-step Reasoning) внутри параметров полей. 

Основные аспекты:
1. **Пошаговое рассуждение (SGR)**: в промпты полей (`creditor`, `debtor_full_name`, `judge_full_name`, `claims_amount`, `grounds`) заложена жесткая методология рассуждения. Модели принудительно заполняют поле `reasoning`, описывая свои выводы по шагам перед выдачей итогового результата.
2. **Извлечение сумм (`claims_amount`)**:
   - Используется агент `agent_claims_amount` со схемой `ClaimsAmountResult`.
   - Модель определяет число обязательств/договоров (`commitments_count`) и список индивидуальных сумм (`amounts`).
   - Исключены любые математические сложения моделью (только выписывание чисел из текста) для предотвращения арифметических ошибок LLM.
3. **Основания требований (`grounds`)**:
   - Извлекаются агентом `agent_grounds` со схемой `GroundsResult`.
   - Модель выбирает строго одно каноническое значение из допустимого списка в промпте (например, `"кредитный договор"`, `"договор займа"`).
   - После получения ответа от LLM в `extraction_agent.py` выполняется дополнительная проверка полученной строки. Если значение не соответствует списку `VALID_GROUNDS`, агент совершает повторную попытку (до 3-х попыток) с указанием допущенной ошибки.

## 7. Специальные обработчики и регуляры (Legacy Overrides)

Для достижения максимального качества и стабильности на чувствительных полях в `extraction_agent.py` встроены жесткие правила-исключения:

### 7.1. Регулярные выражения для судебных решений (`court_decision_ru`)
Для профиля судебных решений извлечение дат и номеров дел перенаправляется на детерминированные регулярные выражения из `court_decision_legacy_rules.py` (чтобы гарантировать 100% точность):
- **Номер дела** (`case_number`)
- **Дата решения** (`decision_date`)
- **Дата окончания процедуры** (`procedure_end_date`)
- **Вычислена ли дата** (`procedure_end_date_is_calculated`)
- **Заблаговременное представление отчета ФУ** (`early_report_deadline`)

### 7.2. Восстановление ИНН и имен кредиторов через веб-поиск (РТК)
- **Для документов ФНС**: 
  - На основе названия инспекции (например, *"Межрайонная ИФНС России № 17 по Новосибирской области"*) запускается веб-поиск ИНН (`search_creditor_inn`).
- **Для не-налоговых документов**: 
  - По извлеченному `creditor_inn` запускается веб-поиск реквизитов организации (`search_creditor_name`).
  - Результаты передаются агенту `company_name_extraction_agent` для нормализации официального наименования (с приведением формы к ООО, ПАО, АО, ПКО и др.) строго по поисковой выдаче, обходя потенциально поврежденный OCR текст документа.

## 5. Валидация полей (validation_service.py)

В системе работает валидатор обязательных полей (`validation_service.validate_fields`).

- **Динамический список**: Обязательные поля определяются динамически на основе YAML-профиля обрабатываемого документа (считываются поля с параметром `required: true` для текущего экстрактора — `fields_nlp` или `fields_llm`).
- **Совместимость**: Если профиль не передан (например, в устаревших тестах), валидатор использует резервный жестко закодированный список полей для MVP (профиль `court_decision`).
