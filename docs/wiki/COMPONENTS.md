# 🧩 Компоненты и Пайплайны

В этом документе описывается, как работают ключевые модули извлечения текста и маршрутизации документов в платформе, а также какие поля мы умеем доставать.

## 1. Пайплайны (Pipeline Engine)

Вся обработка документов является **декларативной**. Когда документ попадает в систему, оркестратор (`orchestration/pipeline_engine.py`) читает его и определяет профиль обработки.

### 1.1 Роутинг и Классификация (system/router.yaml)
Если пользователь загрузил файл, но не указал тип документа, оркестратор запускает LLM-классификатор (через OpenAI/OpenRouter API). **Эвристики для определения типа документа не используются, только LLM.**

1. LLM анализирует извлеченный текст.
2. Возвращает JSON с полем `document_type` и `confidence`.
3. Если уверенность ниже `min_confidence` (указанного в router.yaml), документу принудительно присваивается статус `unknown`.
4. В противном случае `document_type` сопоставляется с одним из профилей (например, `court_decision_ru.yaml` или `rtk.yaml`).

> [!NOTE]
> В базу данных всегда пишется поле `detection_source`, которое указывает, как был определен тип:
> - `request` (передан напрямую при загрузке)
> - `llm` (определен через LLM)
> - `low_confidence_fallback` (уверенность LLM ниже порога)
> - `llm_unavailable` / `llm_no_text` (fallback на `unknown`)

### 1.2 Выполнение Профиля
Каждый профиль содержит:
- **`applicable_sources`**: источники (crm, email, api).
- **`pipeline`**: список шагов (extract_entities → validate → compute_quality).
- **Выбор экстрактора**: ключ `extractor` (со значениями `llm` или `nlp`), который определяет движок для извлечения полей.
- **`fields_llm`** / **`fields_nlp`**: схема полей, которые нужно извлечь.

## 2. Типы документов и извлекаемые поля

Сейчас в системе настроены два основных профиля документов. Для каждого профиля задан строгий набор извлекаемых полей:

### 2.1 Решение суда (`court_decision_ru.yaml`)
- **ФИО должника** (debtor_full_name)
- **ИНН должника** (debtor_inn)
- **Номер дела** (case_number)
- **ФИО судьи** (judge_full_name)
- **Название суда** (court_name)
- **Дата решения** (decision_date)
- **Тип процедуры** (procedure_type) — всегда возвращается в **именительном падеже** (например: `"процедура реализации имущества гражданина"`, а не `"процедуры реализации..."`)
- **Дата окончания процедуры** (procedure_end_date) — *опционально*
- **Вычислена ли дата** (procedure_end_date_is_calculated) — *опционально*
- **Заблаговременное предоставление отчета ФУ** (early_report_deadline) — *опционально*
- **Мотивирующая часть** (motivating_part) — *опционально*
- **Резолютивная часть** (resolutive_part) — *опционально*

### 2.2 Заявление о включении в РТК (`rtk.yaml`)
- **Кредитор** (creditor): Наименование юридического или ФИО физического лица.
- **Сумма требований** (claims_amount): Общая сумма минус госпошлина (если она включена).
- **Основания** (grounds): Договоры, судебные решения, акты, накладные и др. (см. раздел 6.2 про `GROUNDS_PATTERNS`).
- **Номер дела** (case_number): Номер дела Арбитражного суда в формате `А05-6/2025`.

**Особенности для налоговых документов (ФНС):**
- Система автоматически определяет документы ФНС по ключевым фразам (МИНФИН, МЕЖРАЙОННАЯ ИНСПЕКЦИЯ).
- **Порядок экстракции**: сначала `creditor`, потом `creditor_inn` (обратный порядок по сравнению с обычным РТК).
- **Кредитор**: извлекается `creditor_header` (наименование и номер межрайонной инспекции) через специализированный SGR-промпт.
- **ИНН кредитора**: принудительно ищется через веб-поиск (`search_creditor_inn`) по имени межрайонной инспекции для защиты от галлюцинаций LLM.

## 3. Модуль Извлечения Текста и OCR (ocr_service.py)

Служит "входной дверью" для любых файлов. Выполняется один раз в начале обработки.

1. **PDF с текстовым слоем**: Парсится через библиотеку `pdfplumber`. Это самый быстрый и точный метод.
2. **Сканированные PDF или Картинки**: Если текстовый слой отсутствует, запускается OCR-движок. В системе реализован **каскадный механизм (fallback)**, который управляется через переменную окружения `OCR_ENGINE`:
   - **DeepSeek OCR** (`engine="deepseek"`): Запрос отправляется на удаленный сервер (через Ollama API) для высококачественного распознавания VLM-моделью.
   - **Tesseract OCR** (`engine="tesseract"`): Легкий локальный OCR (tesseract-ocr-rus). Является движком по умолчанию.

### 3.1. DeepSeek OCR: retry и таймаут

При использовании DeepSeek OCR на каждую страницу документа выполняется до `OCR_DEEPSEEK_PAGE_RETRIES` (по умолчанию 3) попыток с exponential backoff (2^attempt секунд). Таймаут на одну попытку задаётся через `OCR_DEEPSEEK_TIMEOUT_SECONDS` (по умолчанию 500 секунд).

Логика отката:
- Страница → retry (до N раз) → если всё ещё ошибка → **весь DeepSeek OCR падает**
- DeepSeek OCR упал → **Tesseract** (весь документ целиком)

> [!IMPORTANT]
> Fallback на Tesseract происходит **только после исчерпания всех retry** для одной страницы. Это гарантирует, что DeepSeek получит максимальное число попыток, прежде чем система переключится на менее качественный локальный OCR.

## 4. Механизмы экстракции (LLM vs NLP)

В YAML-профилях через ключ `extractor` можно выбирать, как именно извлекать данные из текста.

### 4.1. LLM Экстракция (`extractor: llm`)
- Использует облачные или удаленные LLM (включая локальные модели через Ollama API).
- Основная модель: `openai/gpt-oss-20b` (с поддержкой Structured Outputs).
- **Схемы и PydanticAI**: Экстракция выполняется модулем `extraction_agent.py` через фреймворк **PydanticAI**. Для каждого поля (creditor, claims_amount, grounds и т.д.) созданы специализированные агенты с привязкой к конкретным Pydantic-моделям (например, `CreditorResult`, `ClaimsAmountResult`). Это позволяет LLM гарантированно возвращать валидный JSON без необходимости ручного парсинга (устаревшие методы с `ast.literal_eval` удалены).
- **Fallback цепочка**: В случае недоступности основной модели, запрос автоматически переходит к `openai/gpt-oss-120b`, а затем к `x-ai/grok-4.1-fast`.
- Использует схемы описания из блока `fields_llm` конфигурации.

### 4.2. Локальный NLP-экстрактор (`extractor: nlp`)
Микросервис `NLP-entity-extractor`, который оформлен как библиотека и работает локально на CPU без обращения к внешним API.
Использует схемы из блока `fields_nlp`.

Внутри проекта используются **несколько обученных локальных моделей** (в `models/exports/`):

1. **Модель NER (Поиск сущностей)**:
   - `bootstrap-russian-ner`
   - Архитектура: Transformer Token Classifier (на базе `Gherman/bert-base-NER-Russian`).
   - Ищет ФИО, ИНН, названия судов, даты и номера дел.
   - Использует `pymorphy3` для приведения ФИО к именительному падежу.
   - **ВАЖНО**: NER является `fallback`-источником для ФИО. Основным источником теперь является Ollama LLM (см. раздел 6.1).

2. **Модель классификации (Early Report Classifier)**:
   - `early-report-classifier`
   - Архитектура: Binary Sequence Classifier.
   - Отвечает за бинарную классификацию специфических полей: определяет наличие или отсутствие требования о "Заблаговременном предоставлении отчета ФУ".

3. **Резервный поиск (Regex Fallback)**:
   - В экстраторах `rtk_nlp` и `court_decision_nlp` встроены резервные механизмы поиска через регулярные выражения. Если основной запрос к Ollama возвращает `None` или падает по таймауту/ошибке, система автоматически переключается на локальные регулярные выражения.

## 5. Валидация полей (validation_service.py)

В системе работает валидатор обязательных полей (`validation_service.validate_fields`).

- **Динамический список**: Обязательные поля определяются динамически на основе YAML-профиля обрабатываемого документа (считываются поля с параметром `required: true` для текущего экстрактора — `fields_nlp` или `fields_llm`).
- **Совместимость**: Если профиль не передан (например, в устаревших тестах), валидатор использует резервный жестко закодированный список полей для MVP (профиль `court_decision`).

## 6. Локальный Ollama LLM — SGR экстракция

Для задач, требующих понимания контекста (а не просто поиска паттернов), система использует **локальную LLM через Ollama API** с методологией **SGR (Step-by-step Reasoning — пошаговое рассуждение)**. Ollama URL настраивается через переменную окружения `OCR_OLLAMA_OCR_URL` (по умолчанию `http://localhost:11434`). Модель: `gpt-oss:20b`.

> [!IMPORTANT]
> Все вызовы к Ollama имеют встроенный **fallback**: при любой ошибке (таймаут, 502, ошибка JSON) система продолжает работу, переключаясь на regex/NER. Это обеспечивает отказоустойчивость.

### 6.1 ФИО судьи и должника (`court_decision_nlp`)

**Файл**: `src/ocr_platform/profiles/court_decision_nlp/rules.py` → `extract_fio_with_ollama_llm()`

**Логика работы** в `transformer.py → predict()`:
1. Вызывается `extract_fio_with_ollama_llm(text)` **до** запуска Transformer NER.
2. Ollama получает **шапку документа (первые 3 000 симв.) + конец документа (последние 2 000 симв.)**, чтобы видеть и заголовок («в составе судьи») и подпись внизу («Судья А.В. Агаларова»).
3. Модель возвращает JSON: `{"judge_fio": "...", "debtor_fio": "...", "reasoning": "..."}`.
4. Если Ollama вернула результат — он **заменяет** результат NER.
5. Если Ollama недоступна/вернула ошибку — NER остаётся в силе (fallback).

**Вспомогательная функция** `_fio_str_to_components(fio_str)` в `transformer.py`:
- Парсит строку «Фамилия Имя Отчество» или «Фамилия И.О.» в объект `FioComponents`.
- Поддерживает формат с инициалами: `"Агаларова А.В."` → `FioComponents(last_name="Агаларова", first_name="А.", patronymic="В.")`.

> [!NOTE]
> **Почему нужен Ollama для ФИО**: Transformer NER не справляется с:
> - Усечёнными словами из-за OCR («Александров» вместо «Александровна»).
> - OCR-опечатками в фамилиях («Буры» вместо «Буров»).
> - Выбором правильной роли, когда в документе есть только одна шапка и подпись в разных частях (берёт ФИО должника вместо судьи).

### 6.2 Наименование кредитора (`rtk_nlp`)

**Файл**: `src/ocr_platform/profiles/rtk_nlp/rules.py` → `extract_creditor_with_ollama_llm()`

- Обрабатывает до **30 000 символов** документа РТК.
- SGR-промпт: ищет наименование кредитора по всему тексту (не только в шапке), выбирает наиболее часто встречающийся вариант написания.
- Fallback: `CREDITOR_BLOCK_RE` (регулярное выражение по шапке «Кредитор:»).

**Причина** использования Ollama вместо только regex: наименование кредитора в шапке может быть плохо распознано OCR (например, «ПАО Сбербан»), тогда как в тексте оно встречается несколько раз в правильной форме («ПАО Сбербанк»).

### 6.3 Сумма требований (`rtk_nlp`)

**Файл**: `src/ocr_platform/profiles/rtk_nlp/rules.py` → `extract_claims_amount_with_ollama_llm()`

- Обрабатывает последние **6 000 символов** (просительная часть «ПРОШУ:»).
- Возвращает JSON с полями:
  - `total_debt_amount` — общая сумма задолженности (float).
  - `is_duty_included_in_total` — включена ли госпошлина в сумму (bool).
  - `duty_amount` — сумма госпошлины (float).
  - `commitments_count` — количество обязательств/кредитных договоров.
  - `reasoning` — пошаговые рассуждения.
- Если `is_duty_included_in_total == true` — госпошлина вычитается из суммы.
- Fallback: regex-поиск по блоку «ПРОШУ:».

## 7. Паттерны оснований (`GROUNDS_PATTERNS`) в `rtk_nlp`

**Файл**: `src/ocr_platform/profiles/rtk_nlp/rules.py`

Список `GROUNDS_PATTERNS` содержит регулярные выражения для 16+ целевых оснований. Порядок паттернов **критически важен** — первое совпадение выигрывает.

> [!CAUTION]
> **Порядок паттернов нельзя произвольно менять.** Более специфичные паттерны должны стоять **выше** более общих, иначе возникают ложные срабатывания.

**Актуальный порядок (от специфичного к общему)**:

| # | Паттерн | Канонический результат |
|---|---------|----------------------|
| 1 | `договор на предоставление коммунальных услуг` | договор на предоставление коммунальных услуг |
| 2 | `договор предоставления кредита` | кредитный договор |
| 3 | `договор потребительского микрозайма` | договор потребительского микрозайма |
| 4 | `договор потребительского займа` | договор потребительского займа |
| 5 | `договор кредитной карты` | кредитный договор |
| 6 | `договор банковского счета` | договор банковского счета |
| 7 | `договор энергоснабжения` | договор энергоснабжения |
| **8** | **`кредитный договор`** | **кредитный договор** ← должен быть ДО займа |
| **9** | **`договор займа`** | **договор займа** |
| 10 | `обязанность по уплате налога` | задолженность по уплате налога |
| 11 | `задолженность по уплате налога` | задолженность по уплате налога |
| 12 | `налоговая задолженность` | налоговая задолженность |
| 13 | `исполнительный лист` | исполнительный лист |
| 14 | `исполнительный документ` | исполнительный документ |
| 15 | `судебный приказ` | судебный приказ |
| 16 | `судебный акт` | судебный акт |
| 17 | `административное правонарушение` | административное правонарушение |

> [!NOTE]
> **Важно**: Паттерн «кредитный договор» (#8) стоит **перед** «договор займа» (#9). Это необходимо, потому что в кредитных документах иногда встречается слово «займа» в контексте (например, сравнение договоров), что приводило к ложному срабатыванию паттерна займа.

> [!NOTE]
> **Налоговые документы**: Для ФНС-документов добавлен паттерн «обязанность по уплате налога» (#10), который срабатывает на фразу типа «обязанности по уплате налогов, сборов» — это типичная формулировка в налоговых РТК.
