# Платформа для OCR-обработки и Интеллектуального Анализа Документов

Комплексное решение для автоматического распознавания (OCR) и глубокого лингвистического анализа (NLP) судебных решений и других официальных документов. 

Система позволяет принимать файлы из различных источников, извлекать текстовые слои, классифицировать типы документов и вытаскивать структурированные сущности (ФИО, ИНН, номера дел и даты) с высоким качеством и встроенным конвейером мониторинга.

---

## 📚 База Знаний (LLM Wiki) и Работа с AI-Агентами

Вся бизнес-логика, архитектурные слои, типы документов и правила маршрутизации задокументированы в формате **LLM Wiki** (по паттерну Андрея Карпаты) в директории `docs/wiki/`. 

Эта папка является **Единым источником истины (Single Source of Truth)** для всех участников проекта.

**Инструкция по настройке AI-агентов (Antigravity, Cursor, Claude):**
При начале работы на новой машине, сервере или в новой сессии, **ОБЯЗАТЕЛЬНО** проинструктируйте вашего агента ознакомиться с этой базой знаний.

Например, если вы используете **Antigravity**, напишите ему в чат:
> *"Создай базовый Knowledge Item: всегда перед планированием любых задач изучай оглавление папки `docs/wiki/` (в частности `AGENT_RULES.md` и `ARCHITECTURE.md`) и придерживайся описанных там правил. После выполнения крупных задач — обновляй соответствующие Markdown-файлы."*

Агент сохранит это в свою долгосрочную память (Knowledge Items), и ваша платформа всегда будет поддерживаться в архитектурной чистоте.

## 🏛 Архитектура проекта

Проект построен по модульному принципу и разделен на два ключевых компонента:

```text
OCR/ (корень проекта)
├── OCR-document-parser/   # Основная платформа, оркестратор и OCR
└── NLP-entity-extractor/  # (Ранее NLP_fio_reader) Локальный NLP/NER микросервис
```

### 1. OCR-document-parser (Ядро и Конвейер)
Это центральный оркестратор системы, написанный на **FastAPI + PostgreSQL**. Он отвечает за бизнес-логику верхнего уровня и инфраструктуру:
* **Извлечение текста:** Работает с "чистыми" PDF (извлечение текстового слоя) и со сканами/изображениями (применяет **Tesseract OCR**).
* **Маршрутизация (Routing):** На основе декларативных YAML-профилей определяет, по какому пайплайну пустить конкретный документ.
* **Классификация:** Автоматически определяет тип входящего документа с использованием LLM-моделей.
* **Очереди задач:** Асинхронный прием документов (Ingest) через **RabbitMQ**.
* **Качество и верификация:** Вычисляет оценку качества обработки (Quality Score) и отправляет подозрительные результаты в очередь ручной проверки (Human Review).
* **Наблюдаемость:** Интегрирован со стеком **Loki + Grafana** для логов и **MLflow** для отслеживания экспериментов.

### 2. NLP-entity-extractor (Интеллектуальный Анализ)
Специализированный локальный микросервис, сфокусированный на работе с русскоязычным текстом с помощью нейросетей:
* **Извлечение полей (NER):** Использует предобученную на русском языке BERT-модель (`Gherman/bert-base-NER-Russian`) для поиска специфических сущностей: ФИО заявителя/должника, ФИО судьи, название суда, номер дела, ИНН и ключевые даты.
* **Морфологическая обработка:** Автоматически нормализует извлеченные имена и сущности (приводит ФИО к именительному падежу с помощью библиотеки `pymorphy3`).
* **Maintanance & Fine-tuning:** Содержит встроенные CLI-инструменты для полуавтоматической разметки датасетов (bootstrap-labeling) и дообучения локальной модели под специфику ваших документов.

## 🔄 Жизненный цикл обработки документа

Взаимодействие компонентов происходит по следующему сценарию:

1. **Прием документа:** Внешняя система делает `POST /documents/ingest` в `OCR-document-parser`.
2. **Сбор текста:** Оркестратор сохраняет файл и извлекает текст (считывая PDF-слой или прогоняя страницы через OCR Tesseract).
3. **Роутинг:** Система определяет тип документа (например, решение суда о банкротстве) и подбирает YAML-профиль обработки.
4. **Экстракция данных:** Оркестратор вызывает модуль `document_intel_service`, который передает полученный текст в локальный экстрактор `NLP-entity-extractor`.
5. **Глубокий NLP-анализ:** Экстрактор с помощью Transformer NER находит в тексте все необходимые сущности, нормализует ФИО и возвращает структурированный JSON.
6. **Валидация и сохранение:** Оркестратор сверяет поля (например, валидность ИНН), формирует Quality Score и записывает финальный результат в PostgreSQL.
7. **Мониторинг:** Все метрики (время работы, уверенность модели, заполненность полей) отправляются в MLflow для последующей аналитики.

## 🛠 Технологический стек

* **Backend API:** Python, FastAPI, Uvicorn, Pydantic
* **Базы данных и хранилища:** PostgreSQL (SQLAlchemy), S3/Local file storage
* **Очереди сообщений:** RabbitMQ
* **OCR & Парсинг:** Tesseract OCR, pdfplumber, pypdf, pdf2image
* **Машинное обучение (NLP):** PyTorch, HuggingFace Transformers (BERT), pymorphy3
* **Observability & MLOps:** MLflow, Loki, Promtail, Grafana

## 📂 Структура каталогов

* `/OCR-document-parser` — Исходный код основной платформы (API, Оркестрация, Storage).
* `/NLP-entity-extractor` — Исходный код локального NER-экстрактора и пайплайны обучения.
* `/data` — Директория для локального хранения файлов документов и логов.
* `/mlruns` и `mlflow.db` — Локальная база данных экспериментов MLflow.
* `/диаграммы` — Архитектурные схемы и диаграммы процессов.
* `/документы` — Исходные образцы файлов для тестирования системы.
