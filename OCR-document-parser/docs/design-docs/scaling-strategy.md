# Стратегия масштабирования OCR-платформы

Документ описывает поэтапный подход к масштабированию при росте нагрузки: 10+ одновременных документов, 7–10 профилей с разными настройками LLM и NLP.

---

## Уровень 1: Горизонтальное масштабирование (сейчас)

**Действия:**
- Запустить 10+ воркеров.
- Добавить rate limiting в `llm_gateway` (семафор/очередь).
- Мониторить память и CPU на воркер.

**Ограничение:** при росте числа NLP-профилей память на воркер будет расти.

---

## Уровень 2: Двухфазный пайплайн (если память станет проблемой)

Профиль известен только после OCR и определения типа. Разделение пайплайна на две фазы:

```
[Ingest] → Queue 1 (ocr.ingest)
    ↓
[Worker Phase 1] OCR + detect_document_type → (text, profile_id)
    ↓
[Queue 2] ocr.extract.{profile_id}  (отдельные очереди по профилю)
    ↓
[Worker Phase 2] Extract + validate + quality
```

**Плюсы:**
- Воркеры Phase 2 могут быть специализированы: NLP-воркеры подписываются только на NLP-очереди и загружают только нужные модели.
- LLM-воркеры — без тяжёлых моделей.
- Масштабирование по типу нагрузки.

**Минусы:**
- Усложнение: две очереди, два типа воркеров, роутинг.
- Дополнительная сериализация/десериализация между фазами.

---

## Уровень 3: Вынос OCR в отдельный сервис

OCR — stateless, CPU-bound, без тяжёлых моделей.

```
[Worker] → HTTP/gRPC → [OCR Service] → text
```

**Плюсы:**
- OCR масштабируется по CPU.
- Воркеры становятся легче (без Tesseract, pdf2image и т.п.).
- Простая интеграция.

**Минусы:**
- Сетевой вызов и сериализация.
- Дополнительный сервис в инфраструктуре.

---

## Уровень 4: Вынос NLP inference в отдельный сервис

NLP-модели живут в одном (или нескольких) inference-сервисах.

```
[Worker] → HTTP/gRPC → [NLP Inference Service] → extracted fields
```

**Плюсы:**
- Один экземпляр модели на сервис вместо 10 копий на 10 воркерах.
- Масштабирование inference по GPU/CPU.
- Воркеры без PyTorch — меньше памяти и проще деплой.

**Минусы:**
- Сетевой вызов.
- Нужно версионирование моделей и A/B-тесты.

---

## Сводная таблица

| Уровень | Триггер | Действие |
|---------|---------|----------|
| 1 | Сейчас | 10 воркеров, rate limiting в LLM |
| 2 | RAM на воркер > 8–10 GB | Двухфазный пайплайн (OCR+роутинг → extract по профилю) |
| 3 | CPU OCR становится bottleneck | Вынос OCR в отдельный сервис |
| 4 | 5+ профилей с NLP, память критична | Вынос NLP inference в отдельный сервис |

# Упрощение первичной обработки (ранняя маршрутизация)

Сейчас все документы идут в одну очередь `ocr.ingest`. Профиль определяется только после OCR и LLM-определения типа. Рекомендации по сокращению нагрузки за счёт параметров запроса и маршрутизации по источнику.

### Что уже есть

- **`document_type` в запросе** — LLM для определения типа не вызывается (роутер пропускает детекцию).
- **`content_type`** — для `text` OCR не нужен; для `image` — всегда OCR; для `pdf` — неизвестно до проверки текстового слоя.

### Новые параметры запроса

| Параметр | Смысл | Эффект |
|----------|-------|--------|
| `force_ocr=true` | «Это скан, сразу OCR» | Пропуск pdfplumber/pymupdf для PDF, экономия CPU |
| `skip_ocr=true` | «Есть текстовый слой» | При пустом тексте не вызывать OCR |

### Маппинг source_type → document_type

В `router.yaml`:

```yaml
# Если document_type не передан — взять из источника
source_type_to_document_type:
  portal: court_decision
  crm: court_decision
  email: court_decision
  external: null   # неизвестно, нужен LLM
  other: null
```

**Эффект:** если `document_type` пустой, а `source_type` есть в маппинге — использовать его вместо LLM.

### Маппинг source_type → OCR-подсказка

```yaml
source_type_to_ocr_hint:
  portal: force_ocr      # портал часто присылает сканы
  crm: try_text_first
  email: try_text_first
```

### Ранняя маршрутизация по очередям

При наличии `document_type` и `content_type` профиль можно определить на этапе ingest (до worker):

| Условие | Очередь | Что не делаем |
|---------|---------|----------------|
| `document_type` известен + `content_type=text` | `ocr.ingest.light` | OCR, LLM для типа |
| `document_type` известен + `content_type=image` | `ocr.ingest.ocr` | LLM для типа |
| `document_type` известен + `content_type=pdf` + `force_ocr=true` | `ocr.ingest.ocr` | LLM для типа, pdfplumber/pymupdf |
| `document_type` известен + `content_type=pdf` + `force_ocr=false` | `ocr.ingest.medium` | LLM для типа |
| `document_type` неизвестен | `ocr.ingest.full` | — |

Воркеры могут быть специализированы: **light** (без OCR/NLP), **ocr** (тяжёлые), **full** (текущий пайплайн).

### Оценка снижения нагрузки

| Оптимизация | Экономия |
|-------------|----------|
| `document_type` в запросе | 1 вызов LLM (~1–3 сек) |
| `source_type` → `document_type` | 1 вызов LLM, когда источник однозначен |
| `force_ocr=true` для сканов | 2 попытки pdfplumber/pymupdf (~0.5–2 сек) |
| `skip_ocr=true` для PDF с текстом | Пропуск OCR при пустом тексте |
| Очереди light/ocr/full | Разделение нагрузки, меньше тяжёлых воркеров |

### Порядок внедрения

1. Добавить `force_ocr` / `skip_ocr` в схему запроса и использовать в `extract_text_at_ingest`.
2. Добавить `source_type_to_document_type` в router и использовать при отсутствии `document_type`.
3. Реализовать раннее определение `profile_id` в API и маршрутизацию по очередям.
4. Опционально: `source_type_to_ocr_hint` как fallback, когда `force_ocr`/`skip_ocr` не заданы.
