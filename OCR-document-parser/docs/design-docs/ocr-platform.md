# OCR‑платформа для судебных документов — дизайн

Этот документ конкретизирует архитектуру (`ARCHITECTURE.md`) применительно к задачам OCR и извлечения данных из судебных решений.

Основные цели:

- единая платформа для разных типов документов;
- декларативные профили пайплайнов;
- воспроизводимость и трассируемость обработок;
- встроенная система оценок качества и маршрутизации на ручную проверку;
- возможность гибридной работы с облачными и локальными моделями.

---

## Домены и пользовательские сценарии

### Источники документов

Документы поступают из разных систем:

- CRM;
- email;
- личный кабинет;
- внешние интеграции (API других систем).

Каждый входящий запрос описывается:

- `source_type` — тип источника;
- `document_type` (опционально) — явный тип документа от источника (например, `court_decision`);
- `document_type_hint` (опционально) — legacy-подсказка для обратной совместимости;
- содержимым (PDF/изображение);
- метаданными (идентификаторы во внешних системах, язык, дата и т.д.).

### Основной сценарий (MVP)

1. Клиент отправляет документ через `POST /documents/ingest`.
2. API сохраняет документ и инициирует запуск пайплайна.
3. Оркестратор **сначала извлекает текст** (см. раздел «Извлечение текста на входе»), затем выбирает профиль.
4. Пайплайн выполняет:
   - сохранение оригинала;
   - извлечение полей из уже полученного текста;
   - валидацию полей;
   - оценку качества;
   - решение о ручной проверке.
5. Клиент получает результат через `GET /documents/{id}/result`.

### Извлечение текста на входе пайплайна

Текст извлекается **один раз в начале** обработки, до определения типа документа и до экстракции полей. Это обеспечивает:

- **PDF с текстовым слоем** — используется `pdfplumber` (без OCR).
- **PDF без текста (скан)** — OCR (pdf2image + Tesseract) по страницам.
- **Изображение** — OCR (Tesseract).
- **Текстовый файл** — чтение содержимого.

Сценарии:

1. **Документ без типа, PDF/скан без текста**  
   Извлечение текста → OCR → текст для LLM-определения типа → профиль → экстракция полей.

2. **Документ с типом, PDF/скан без текста**  
   Извлечение текста → OCR → профиль по `document_type` → экстракция полей.

3. **Документ с текстовым слоем**  
   Извлечение текста без OCR → определение типа / экстракция полей.

Функция `extract_text_at_ingest` в `ocr_service` возвращает `(текст, ocr_was_used)`. Флаг `ocr_was_used` логируется в MLflow.

## Структура пакета `ocr_platform`

Слои и их ответственность заданы в `ARCHITECTURE.md`. Здесь — привязка к конкретным задачам OCR.

### API

Сервис предоставляет как минимум следующие эндпоинты:

- `POST /documents/ingest`
  - вход:
    - `source_type`, `document_type_hint`;
    - `content_type` (`pdf`, `image`);   
    - `meta` (JSON).
  - выход:
    - `document_id`;
    - `pipeline_run_id`;
    - `status`.
- `GET /documents/{id}/result`
  - возвращает:
    - последнюю версию распознанного текста;
    - последнюю версию структурированных данных;
    - оценки качества;
    - статус валидации;
    - флаг и причину ручной проверки.
- `GET /human-review/tasks`, `POST /human-review/tasks/{id}/submit`
  - работа с очередью ручной проверки.

### Оркестратор

Оркестратор:

- мапит (`source_type`, `document_type`) → `profile_id`;
- если `document_type` не передан, запускает LLM-классификацию типа документа;
- при недоступности/ошибках LLM или низкой уверенности выбирает тип `unknown`;
- читает YAML‑профиль (например, `court_decision_ru.yaml`);
- исполняет шаги в указанном порядке;
- фиксирует события шагов в БД и логах.

Каждый шаг:

- имеет уникальное имя (например, `store_original`, `detect_document_type`, `extract_entities`, `validate_entities`, `compute_quality`, `route_to_human_review`);
- получает и возвращает объект контекста (например, `PipelineContext`), включающий ссылки на:
  - `document_id`, `pipeline_run_id`;
  - промежуточные артефакты (пути к файлам, текст, структурированные данные и т.д.);
  - накопленные метрики, промпты и флаги.

### Сервисы

#### OCR‑сервис

Функции:

- `extract_text_at_ingest(file_path, content_type)` — единая точка извлечения текста на входе:
  - PDF: текстовый слой (pdfplumber) → при пустоте OCR;
  - Image: OCR (Tesseract);
  - Text: чтение файла.
  - Возвращает `(текст, ocr_was_used)`.
- `extract_text_from_pdf` — извлечение текстового слоя PDF через pdfplumber (без OCR).
- `run_ocr(file_path, content_type)` — внутренняя функция OCR через Tesseract (используется внутри `extract_text_at_ingest`):
  - PDF: pdf2image → изображения страниц → Tesseract (rus+eng);
  - Image: Tesseract по одному изображению.
- `run_ocr_if_needed` — внутренняя обёртка: возвращает existing_text, если не пуст; иначе вызывает `run_ocr`.

Зависимости: `pdf2image` (poppler), `pytesseract` (tesseract-ocr, tesseract-ocr-rus).

#### Document Intelligence

Функции:

- `simple_extract_fields` — единая точка входа для экстракции полей:
  - диспетчеризует по `profile_id` в реестр обработчиков (`ocr_platform.profiles`);
  - вызывает соответствующий `ProfileHandler.extract(...)`.
- Профили возвращают «чистый результат»; сохранение, метрики и MLflow остаются в `services` и `orchestration`.
- Умеет работать по профилю:
  - какие поля требуются;
  - какие есть альтернативные источники (LLM, локальный NLP, правила).

### Профили (`profiles/`)

Каждый профиль — отдельный пакет с обработчиком:

- `profiles/__init__.py` — реестр `profile_id → ProfileHandler`, диспетчеризация по `extractor`;
- `profiles/base.py` — протокол `ProfileHandler`;
- `profiles/court_decision_ru/` — LLM-экстракция (при `extractor: llm`);
- `profiles/court_decision_nlp/` — NLP-экстракция (при `extractor: nlp` в court_decision_ru);
- `profiles/unknown/` — без экстракции полей.

Правила:

- `services` вызывает `profiles`, а не наоборот;
- профили не обращаются напрямую к БД, логам, API;
- профили возвращают `Dict[str, dict]` с полями `reasoning`, `value`, `confidence`.

#### Validation

Функции:

- проверка форматов:
  - ИНН, номера дела, дат, ФИО и т.д.;
- проверка допустимых диапазонов:
  - дата завершения процедуры не раньше даты начала и т.п.;
- согласованность:
  - одно и то же значение (например, номер дела, фамилия имя) должно совпадать в разных местах;
- интеграция со справочниками:
  - список судов, типы процедур и т.д.

Результат:

- `validation_status` — `ok | warnings | errors`;
- список `validation_issues` с кодами и описаниями.

#### Quality

Функции:

- оценка технического качества:
  - анализ изображений/страниц (резолюция, шум, размытость, обрезка, пустые области, размер текста);
- интеграция с семантической уверенностью и валидацией;
- формирование `technical_quality_score`, `semantic_confidence_score`, `overall_quality_score` и флага `human_review_required`.

Подробности формул и порогов — в `QUALITY_SCORE.md` и YAML‑профилях.

## Профили пайплайнов и глобальный роутинг

Конфиги живут в:

- `src/ocr_platform/config/pipelines/system/router.yaml` — глобальный роутинг типа документа в профиль;
- `src/ocr_platform/config/pipelines/profiles/*.yaml` — профили конкретных типов документов.

Профили описывают:

- `profile_id`;
- `extractor` (для court_decision_ru: `llm` или `nlp`, по умолчанию `llm`);
- `applicable_sources`;
- `pipeline` (шаги и условия ветвления);
- `models` (какие OCR/LLM использовать);
- `thresholds` (пороги качества и уверенности);
- `fields_llm` / `fields_nlp` (отдельные наборы полей для каждого экстрактора).

### Примеры профилей

**court_decision_ru** — один профиль, выбор экстрактора через `extractor: llm | nlp`:

- `extractor: llm` — LLM-экстракция (поля из `fields_llm`);
- `extractor: nlp` — локальная NLP-экстракция (поля из `fields_nlp`);
- шаги:
  - `store_original` — сохранить входной файл;
  - `detect_document_type` — уточнить тип документа (текст уже извлечён на входе);
  - `extract_entities` — извлечь поля (ФИО, ИНН, номер дела и т.д.);
  - `validate_entities` — проверить поля;
  - `compute_quality` — посчитать оценки качества;
  - `route_to_human_review` — принять решение о ручной проверке.
- поля (одинаковые для обоих экстракторов):
  - `debtor_full_name`, `debtor_inn`, `case_number`, `judge_full_name`,
  - `court_name`, `decision_date`, `procedure_type`, `procedure_end_date`.

**unknown** — fallback без экстракции полей.

## Наблюдаемость и MLflow

Каждый шаг пайплайна:

- пишет структурные логи с основными параметрами и результатами;
- обновляет метрики через `observability/metrics.py`;
- для ML/LLM‑шагов:
  - создаёт или использует MLflow run;
  - логирует:
    - параметры моделей и конфигурацию порогов;
    - агрегированные метрики качества;
    - теги с `profile_id`, `pipeline_run_id`, `document_type`, `source_type`.

Для пайплайна дополнительно логируется метрика `ocr_used` (1.0, если был выполнен OCR).

Для LLM-классификации типа документа дополнительно логируются:

- модели из fallback-цепочки;
- модель, давшая итоговый результат (`winner_model`);
- итоговая уверенность (`final_confidence`);
- артефакт со статусами всех попыток (`document_type_attempts.json`).

Это позволяет:

- анализировать качество по профилям и моделям;
- сравнивать разные конфигурации;
- отлаживать деградации.

## Human Review и цикл улучшений

Система ручной верификации:

- получает задачи с документами с низким/пограничным качеством;
- даёт оператору:
  - увидеть оригинал документа и распознанные поля;
  - скорректировать значения;
  - указать тип ошибки и причину исправления.

Каждое исправление:

- создаёт новую версию структурированных данных;
- может использоваться как разметка для:
  - улучшения правил;
  - настройки промптов;
  - обучения/дообучения моделей.

В будущем возможно:

- автоматическая генерация предложений по улучшению правил на основе накопленных исправлений;
- подключение дополнительных моделей для активного обучения.
