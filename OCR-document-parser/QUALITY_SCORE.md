## Система оценок качества обработки документа

Цель этого документа — задать единые правила для:

- расчёта технического и семантического качества;
- принятия решения об автоматическом принятии результата или отправке на ручную проверку;
- логирования метрик в БД, системах мониторинга и MLflow.

Все числовые оценки задаются в диапазоне **[0.0; 1.0]**, где 1.0 — максимально возможное качество/уверенность.

---

## Компоненты качества

### 1. Техническое качество (`technical_quality_score`)

Оценивается **по страницам** и агрегируется по документу.

Признаки на уровне страницы:

- `resolution_dpi` — оценка DPI исходного изображения/страницы PDF.
- `is_low_resolution` — флаг, если DPI ниже порога (например, < 200 DPI).
- `noise_level` — оценка уровня шума.
- `blur_score` — степень размытия (например, по вариации Лапласа).
- `focus_shift` — индикатор сдвига фокуса.
- `empty_area_ratio` — доля пустых областей на странице.
- `cropped_edges` — флаг обрезанных краёв.
- `text_size_score` — оценка читаемости размера шрифта.

Агрегирование по документу:

- вычисляется взвешенная комбинация признаков по всем страницам;
- результат нормируется в диапазон [0;1] и сохраняется как `technical_quality_score`.

### 2. Семантическое качество (`semantic_confidence_score`)

Оценивается по результатам:

- извлечения полей (LLM/правила/справочники);
- валидации (формат, диапазоны, согласованность, внешние проверки).

Признаки:

- `field_confidences` — словарь `field_name → confidence ∈ [0;1]`:
  - включает все ключевые поля (ФИО должника, ИНН, номер дела, ФИО судьи, и т.д.);
  - confidence может формироваться моделью или вычисляться по эвристикам (несколько совпадающих источников и т.д.).
- `validation_status` — `ok | warnings | errors`.
- `validation_issues` — список проблем с указанием полей и типов ошибок.

Агрегирование:

- базовый семантический скор — средне‑взвешенное по `field_confidences`;
- штрафы за:
  - критические ошибки валидации (например, невалидный ИНН, пустой номер дела);
  - отсутствие обязательных полей.
- итоговая величина нормируется в [0;1] и сохраняется как `semantic_confidence_score`.

### 3. Итоговая оценка (`overall_quality_score`)

Итоговая оценка объединяет:

- техническое качество скана;
- семантическую уверенность и результаты валидации;
- дополнительные сигналы (например, успешность прошлых обработок по тому же источнику/профилю).

Рекомендованная формула (может быть уточнена в профиле):

- `overall_quality_score = w_tech * technical_quality_score + w_sem * semantic_confidence_score`,
  где, например, `w_tech = 0.4`, `w_sem = 0.6`.

Конкретные веса и пороги задаются в YAML‑профиле для каждого типа документа.

## Пороги и маршрутизация

Для каждого профиля (например, `court_decision_ru`) в его YAML‑конфигурации задаются:

- `technical_quality_min` — минимальное допустимое техническое качество.
- `semantic_confidence_min` — минимальное допустимое семантическое качество.
- `overall_quality_min_auto_accept` — минимальное значение для автоматического принятия результата без ручной проверки.
- `critical_fields` — список обязательных полей (например, `debtor_full_name`, `debtor_inn`, `case_number` и т.д.).

Правила маршрутизации:

1. Если `overall_quality_score >= overall_quality_min_auto_accept`
   **и** все `critical_fields` заполнены **и** `validation_status == "ok"`:
   - `human_review_required = false`.
2. Если `overall_quality_score` ниже порога или `validation_status == "errors"`:
   - `human_review_required = true`,
   - `human_review_reason` содержит агрегированное описание основных проблем.
3. При `validation_status == "warnings"`:
   - решение зависит от профиля и может задаваться отдельным флагом, например `warnings_require_review: true|false`.

## Логирование и хранение оценок

### 1. В БД

В таблице `quality_scores` для каждого `pipeline_run` хранятся:

- `technical_score` — `technical_quality_score`.
- `semantic_score` — `semantic_confidence_score`.
- `overall_score` — `overall_quality_score`.
- `details` — JSON:
  - покомпонентные показатели по страницам;
  - confidence по полям;
  - ключевые флаги (низкое разрешение, сильный шум и т.д.);
  - использованные пороги и веса.

### 2. В логах и метриках

Каждый шаг, связанный с оценкой качества, должен:

- логировать ключевые значения и флаги (структурные логи);
- обновлять метрики:
  - распределение `technical_quality_score`;
  - распределение `semantic_confidence_score`;
  - распределение `overall_quality_score`;
  - доля документов, отправленных на ручную проверку.

### 3. В MLflow

При использовании ML/LLM для:

- оценки качества изображений;
- извлечения сущностей и полей;

необходимо логировать в MLflow:

- модели и промпты;
- параметры моделей и конфигурации порогов;
- агрегированные метрики качества (в том числе `technical_quality_score`, `semantic_confidence_score`, `overall_quality_score`);
- теги:
  - `profile_id`, `pipeline_run_id`;
  - `document_type`, `source_type`.

## Расширение системы оценок

Если требуется изменить или расширить систему оценок:

1. Обновить этот документ (`QUALITY_SCORE.md`) и соответствующие спецификации профилей в `docs/product-specs`.
2. Обновить YAML‑профили (`src/ocr_platform/config/pipelines/*.yaml`) с новыми порогами/весами/флагами.
3. Обновить реализацию в `quality_service.py` и связанные тесты в `tests/services/test_quality_service.py`.
4. При необходимости добавить новые метрики в `observability/metrics.py` и их логирование.
