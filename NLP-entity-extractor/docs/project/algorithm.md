# Алгоритм извлечения

## Назначение

Этот документ описывает, как работает текущий экстрактор от начала до конца, что именно обучается в NLP‑модели, что извлекается правилами и как обслуживать или расширять пайплайн.

## Текущий охват извлечения

Сервис возвращает следующие поля из текстового PDF с судебным решением:

- `applicant_fio`
- `judge_fio`
- `court_name`
- `case_number`
- `inn`
- `decision_date`
- `procedure_end_date`
- `procedure_type`

## Общий поток

1. API получает байты PDF, закодированные в Base64.
2. Байты PDF декодируются и парсятся в обычный текст.
3. Текст нормализуется.
4. Локальная трансформерная NER‑модель находит связанные с ФИО сущности в тексте.
5. Из найденных сущностей собираются кандидатные группы ФИО.
6. Модуль выбора ролей определяет лучшие `applicant_fio` и `judge_fio`.
7. Извлечённое ФИО нормализуется в именительный падеж.
8. Оставшиеся структурированные поля извлекаются детерминированными правилами.
9. API возвращает один JSON‑ответ с полями, уверенностью и исходным фрагментом текста.

## Карта модулей

- Точка входа API: `src/api/app.py`

- Пайплайн дообучения: `src/training/train.py`

## Подробный поток инференса

### 1. От PDF к тексту

API принимает `pdf_base64`, декодирует его и извлекает текст из текстового PDF.

Важное ограничение:

- OCR не используется.
- Если качество извлечения текста из PDF низкое, качество последующего извлечения тоже снижается.

### 2. Нормализация текста

Перед извлечением пробельные символы нормализуются, чтобы уменьшить шум от особенностей верстки PDF.

Это помогает:

- сопоставлять шаблоны для правил
- строить непрерывные спаны ФИО
- готовить более чистые текстовые фрагменты‑доказательства

### 3. NER‑модель для компонент ФИО

Локальная трансформер‑модель предсказывает токен‑уровневые метки для:

- `LAST_NAME`
- `FIRST_NAME`
- `MIDDLE_NAME`

Модель используется только для извлечения ФИО.

Текущее поведение:

- `applicant_fio` и `judge_fio` зависят от NLP‑модели
- все остальные поля сейчас извлекаются правилами, а не за счёт обученной модели

### 4. Сборка кандидатов

Найденные сущности имени группируются в локальные кандидаты ФИО, если токены находятся достаточно близко друг к другу в тексте.

Кандидат должен содержать как минимум:

- `LAST_NAME`
- `FIRST_NAME`

Если также найдено `MIDDLE_NAME`, кандидат получает более высокий балл полноты.

### 5. Выбор роли

Модель может найти несколько ФИО в одном документе, поэтому система не возвращает вслепую первое совпадение.

Вместо этого она строит оценки кандидатов, используя:

- уверенность модели
- полноту ФИО
- близлежащие ключевые слова
- роль‑специфические якоря
- штрафы за негативный контекст

#### Выбор заявителя

Примеры положительного контекста:

- `заявление`
- `должник`
- `гражданин`
- `признать`
- `банкрот`

Примеры отрицательного контекста:

- `судья`
- `финансовый управляющий`
- `представитель`

#### Выбор судьи

Примеры положительного контекста:

- `судья`
- `в составе судьи`
- `председательствующий`

Примеры отрицательного контекста:

- `заявление`
- `должник`
- `гражданин`

Если на первом шаге заявитель и судья разрешаются в одно и то же ФИО, экстрактор пытается найти более позднего отдельного кандидата для роли судьи.

### 6. Нормализация ФИО

В судебных актах люди часто упоминаются в косвенных падежах, например:

- `Олейникова Юрия Владимировича`

Слой постобработки преобразует извлечённое ФИО в именительный падеж:

- `Олейников Юрий Владимирович`

Особенности реализации:

- морфология обрабатывается с помощью `pymorphy3`
- форма фамилии при необходимости корректируется с учётом пола простыми правилами

Именно эта нормализованная форма возвращается API в полях `normalized`, `last_name`, `first_name` и `patronymic`.

### 7. Правила для извлечения других полей

Не‑ФИО поля извлекаются детерминированными правилами на основе регулярных выражений.

Текущие группы правил:

- `court_name`
- `case_number`
- `inn`
- `decision_date`
- `procedure_end_date`
- `procedure_type`

#### `procedure_end_date`

Это поле вычисляется в два этапа:

1. явная дата, например `сроком до 17 июня 2024 года`
2. производная дата из даты решения плюс указанный срок, например `на срок 6 месяцев`

Если в документе указана только длительность и присутствует корректная `decision_date`, конечная дата рассчитывается автоматически.

## Пайплайн обучения

### Что обучается

Сейчас обучается только модель извлечения ФИО.

Конкретно:

- `applicant_fio`
- `judge_fio`

Остальные поля пока не входят в ML‑пайплайн обучения.

### Этапы обучения

1. Сложить много однотипных PDF в `data/raw/train_pdfs/`.
2. Запустить bootstrap‑разметку, чтобы извлечь текст и предварительно заполнить предложения по ФИО.
3. Вручную просмотреть `bootstrap.jsonl`.
4. Пометить пригодные строки как `approved`.
5. Разбить утверждённые строки на обучающую, валидационную и тестовую выборки.
6. Дообучить модель токен‑классификации.
7. Сохранить обученный артефакт модели.
8. Использовать этот артефакт в API для новых похожих документов.

Важно:

- обучение проводится на датасете, а не по одному документу
- одна обученная модель должна обобщаться на новые похожие решения судов

## Рекомендации по сопровождению

### Если качество по ФИО слабое

Проверять по порядку:

1. насколько чисто извлёкся текст из PDF
2. нашла ли базовая NER‑модель корректные токены имени
3. не выбрал ли скоринг ролей неправильное ФИО среди валидных кандидатов
4. достаточно ли в обучающем наборе примеров нужного типа документов

Вероятные исправления:

- добавить больше размеченных PDF
- переобучить NER‑модель
- подправить якоря ролей или веса контекста в `src/inference/transformer.py`
- улучшить нормализацию в `src/inference/postprocess.py`

### Если не извлекается не‑ФИО поле

Проверить:

1. сырой извлечённый текст
2. соответствующее регулярное выражение в `src/inference/rules.py`
3. не использует ли документ новую формулировку, не покрытую текущими правилами

Вероятное исправление:

- расширить или добавить новый шаблон извлечения в `src/inference/rules.py`
- добавить регрессионный тест на новую формулировку

### Если нужно добавить новое поле

Выберите один из двух путей:

1. правило‑ориентированное извлечение, если формулировка поля в документах стабильна
2. модель‑ориентированное извлечение, если поле зависит от контекста и не ловится надёжно правилами

Рекомендуемый подход по умолчанию:

- для жёстко форматированных метаданных суда предпочитать правила
- использовать ML, когда поле зависит от семантической роли и контекста

## Регрессионная надёжность

Любое изменение логики извлечения должно сопровождаться:

- модульным тестом на новый паттерн или поведение
- по возможности проверкой хотя бы на одном реальном примере

Текущий набор регрессионных тестов располагается в:

- `tests/inference/test_rules_and_roles.py`
- `tests/inference/test_heuristics.py`
- `tests/api/test_app.py`

## Известная архитектурная граница

Текущая архитектура гибридная:

- NLP‑модель для извлечения ФИО и их ролей
- правила для остальных структурированных полей

Это сделано намеренно: разрешение ролей ФИО хорошо выигрывает от обучения на контексте, тогда как большинство остальных полей в судебных решениях по‑прежнему легче и надёжнее обслуживать детерминированными правилами.
